# خطة تطوير ModaPella — فصل الأنظمة الثلاثة

## الفكرة العامة

المشروع سيُقسَّم لـ **3 أنظمة منفصلة** تماماً في الواجهة، لكنها تشترك في **نفس قاعدة البيانات** و**نفس الـ Backend**:

| النظام | المسار | من يصل |
|--------|--------|---------|
| 🌐 الموقع العام | `/` | الزبون (بدون تسجيل دخول) |
| 🖥️ داشبورد الإدارة | `/admin/*` | المدير فقط |
| 🧾 نظام الكاشير | `/cashier/*` | الكاشير + المدير |

> [!IMPORTANT]
> **قاعدة بيانات واحدة** — كل التعديلات اللي تعملها في الداشبورد أو الكاشير بتتحدث على نفس الـ MongoDB فوراً.
> الكاشير لو عمل بيعة، المخزون في الموقع بيتقلل تلقائياً عبر Socket.io اللي موجود بالفعل.

## 🖥️ الجزء الثاني — داشبورد الإدارة (Admin Dashboard)

### شاشة مستقلة تماماً على `/admin`

#### الصفحات والأقسام:

### 1. 📊 Overview — الشاشة الرئيسية
- **4 كروت إحصائية** كبيرة: إجمالي الإيرادات، عدد الطلبات، قيمة المخزون، المنتجات
- **جدول أحدث الطلبات** (أونلاين + أوفلاين مع تمييز بين الاتنين)
- **تحذيرات المخزون** — منتجات وصلت لأقل من 5 قطع
- **حالة الموقع** — نشر / إيقاف بزرار واحد

### 2. 🛍️ إدارة المنتجات
- **جدول كامل** بكل المنتجات مع بحث وفلتر بالفئة
- **إضافة منتج** — نموذج كامل: الاسم، الفئة، الوصف، السعر (بالجنيه)، المخزون، الألوان، المقاسات، رابط الصورة
- **تعديل منتج** موجود (نموذج في modal)
- **حذف / أرشفة** منتج
- **عرض مصغر** للمنتج مع صورته

### 3. 📦 الطلبات
- **كل الطلبات** من الموقع + الكاشير مع تمييز النوع
- **تفاصيل كل طلب** — المنتجات، الكمية، السعر، التاريخ
- **تغيير حالة الطلب** (Pending → Completed / Returned)

### 4. 🎨 إعدادات الموقع
- تعديل نص الـ Hero (العنوان، السطر الوصفي)
- تعديل أزرار الـ CTA (النص + الرابط)
- تشغيل / إيقاف الموقع (Maintenance Mode)
- رسالة الصيانة المخصصة

### 5. 👥 إدارة المستخدمين (Admin فقط)
- قائمة بكل الكاشيرين
- إضافة كاشير جديد (الاسم + الإيميل + الباسورد)
- تعطيل حساب كاشير

## 🧾 الجزء الثالث — نظام الكاشير

### شاشة مستقلة على `/cashier`

#### الأقسام:

### 1. ⚡ نقطة البيع (POS Terminal)
- **بحث عن منتج** بالاسم أو الفئة فوراً
- **اختيار اللون والمقاس** للمنتج المختار
- **الكمية** مع عرض المتوفر
- **سلة البيع** — إضافة أكثر من منتج في فاتورة واحدة
- **إجمالي الفاتورة بالجنيه المصري** (ج.م)
- **زر إتمام البيع** ← يُنشئ Order ويُقلل المخزون فوراً

### 2. 🧾 الفاتورة
- بعد إتمام البيع تظهر **فاتورة احترافية** تحتوي:
  - اسم المحل (ModaPella)
  - التاريخ والوقت
  - رقم الفاتورة (Order ID مختصر)
  - قائمة المنتجات (الاسم، المقاس، اللون، الكمية، السعر)
  - الإجمالي بالجنيه المصري
  - **زر طباعة** (window.print) مع CSS خاص بالطباعة
- إمكانية **حفظ الفاتورة كصفحة HTML** للإرسال

### 3. 📦 المخزن
- عرض كل المنتجات مع المخزون الحالي
- **استلام بضاعة** — زيادة مخزون منتج (+N)
- **تعديل يدوي** للمخزون
- **تنبيهات المخزون المنخفض** باللون الأحمر

### 4. 🔄 المرتجعات
- البحث عن طلب برقمه
- إرجاع الطلب ← يُعيد المخزون تلقائياً
- تسجيل سبب الإرجاع

### 5. 📋 سجل اليوم
- كل المبيعات اللي حصلت النهارده
- إجمالي اليوم بالجنيه المصري

## 🔐 نظام الصلاحيات المحدَّث

```
🌐 زبون عادي   →  / فقط (بدون تسجيل)
🧾 cashier     →  / + /cashier/* فقط
👑 admin       →  / + /cashier/* + /admin/*
```

### التعديلات على الكود:

#### Backend — `middleware/auth.js`
سنضيف **middleware** جديد للتحقق من الدور:
```js
// middleware/requireRole.js
module.exports = (allowedRoles) => (req, res, next) => {
  if (!allowedRoles.includes(req.user.role)) {
    return res.status(403).json({ message: 'Access denied' });
  }
  next();
};
```

#### Backend — routes جديدة
- `GET /api/admin/users` — قائمة المستخدمين (admin فقط)
- `POST /api/admin/users` — إضافة كاشير (admin فقط)
- `GET /api/cashier/today` — مبيعات اليوم
- `GET /api/orders/:id` — تفاصيل طلب واحد بالكامل

#### Frontend — `ProtectedRoute.jsx`
```jsx
// ProtectedRoute الجديد يقبل allowedRoles
<ProtectedRoute allowedRoles={['admin']}>
  <AdminDashboard />
</ProtectedRoute>

<ProtectedRoute allowedRoles={['admin', 'cashier']}>
  <CashierDashboard />
</ProtectedRoute>
```

# المرحلة الثانية: نظام الباركود (Barcode System)

## الفكرة العامة ونظام الأكواد (Smart SKUs)

بناءً على طلبك، سنقوم ببرمجة نظام **يولد كود ذكي (SKU)** تلقائياً لكل منتج جديد يتم إضافته. 

### 💡 كيف سيعمل هذا النظام؟ (أفضل ممارسة في السوق)
بدلاً من حشو السعر واسم الشركة داخل "نص الباركود" نفسه (مما قد يسبب مشاكل لو تغير السعر مستقبلاً)، سنقوم بعمل الآتي:
1. النظام سيولد كود مثل `PAN-1001` (PAN اختصار بنطلون، و 1001 رقمه التسلسلي).
2. عندما يقرأ الكاشير هذا الكود بجهاز الباركود، سيقوم النظام بجلب **كل التفاصيل** من قاعدة البيانات فوراً (اسم الشركة، السعر، المقاسات، الخ).
3. سنضيف خانة جديدة باسم **"الشركة الموردة" (Supplier/Brand)** لكل منتج.

## التغييرات المطلوبة في الكود

### 1. قاعدة البيانات (Backend Models)
#### [MODIFY] `backend/models/Product.js`
- إضافة حقل `sku: { type: String, unique: true }` (سيتم توليده تلقائياً).
- إضافة حقل `supplier: { type: String, default: '' }` (اسم الشركة الموردة أو الماركة).

### 2. واجهة الإدارة (Admin Dashboard)
#### [MODIFY] `frontend/src/pages/admin/AdminProducts.jsx`
- إضافة حقل "الشركة / المورد" في نافذة إضافة المنتج.
- إظهار الكود التلقائي (SKU) بعد إضافة المنتج ليتمكن المدير من طباعته.

### 3. خادم الباك إند (Backend Logic)
#### [MODIFY] `backend/routes/admin.js`
- عند إضافة منتج جديد (POST)، سيقوم السيرفر بقراءة الفئة (مثلاً Dress) وتوليد كود تلقائي `DRE-1001` بناءً على عدد المنتجات السابقة في نفس الفئة.

### 4. واجهة الكاشير (Cashier POS)
#### [MODIFY] `frontend/src/pages/cashier/CashierPOS.jsx`
- عمل Custom Hook `useBarcodeScanner` يستمع لضغطات الكيبورد السريعة (التي يرسلها جهاز الباركود).
- عند التقاط كود (مثلاً PAN-1001) ووراءه زر `Enter`، يبحث النظام عن المنتج بهذا الكود ويضيفه للفاتورة تلقائياً.

# المرحلة الثالثة: الخزنة وتتبع الأموال (Cash Drawer / Safe)

## الفكرة العامة
يحتاج المحل إلى نظام لضبط "حركة الأموال" (Cash Flow) داخل درج الكاشير (الخزنة). ولأن الدفع ليس كاش فقط (هناك Instapay وبطاقات بنكية)، يجب فصل **الرصيد النقدي في الدرج** عن **أموال البنك**، حتى يسهل مطابقة الدرج نهاية اليوم.

### 💡 اقتراحات احترافية لتنظيم الدرج:
1. **تعدد طرق الدفع في الكاشير:** عند إتمام البيع، سيختار الكاشير طريقة الدفع (كاش - Instapay/فيزا). 
   - الكاش: يدخل الدرج الفعلي (تزيد عهدة الكاشير).
   - انستا باي: يدخل حساب البنك (تزيد أرباح اليوم ولكن لا تؤثر على الدرج).
2. **فصل الأرصدة في شاشة الخزنة:** شاشة الخزنة ستعرض رصيدين:
   - `رصيد الدرج النقدي`: (الكاشير يحاسب عليه آخر اليوم).
   - `إجمالي إيرادات البنك`: (لمعرفة حجم مبيعات الفيزا/انستا باي).
3. **تصنيفات المصروفات الجاهزة:** بدلاً من كتابة المصروفات عشوائياً، سنوفر قائمة منسدلة (ضيافة، فواتير كهرباء/ماء، نظافة، نولون شحن، دفع لمورد، أخرى) لتنظيف التقارير.
4. **تقفيل الوردية (End of Shift):** زر يضغطه الكاشير آخر اليوم يطبع "تقرير التقفيل" (كم كاش متوقع، كم مصروفات، كم انستا باي) ويصفي الدرج ليبدأ اليوم التالي برصيد فكة جديد.

## التغييرات المطلوبة في الكود

### 1. قاعدة البيانات (Backend Models)
#### [NEW] `backend/models/Transaction.js`
إنشاء موديل جديد `Transaction` يحتوي على:
- `amount` (المبلغ)
- `type` (إما `IN` أموال داخلة أو `OUT` أموال خارجة)
- `category` (نوع العملية: `Sale` مبيعات، `Refund` مرتجع، `Expense` مصروف، `Deposit` إيداع)
- `paymentMethod` (طريقة الدفع: `Cash` كاش، `Instapay` انستاباي/فيزا)
- `description` (وصف أو سبب العملية)
- `user` (الذي قام بالعملية)

#### [MODIFY] `backend/models/Order.js`
- إضافة حقل `paymentMethod` للأوردر وتمريره عند البيع.

### 2. الباك إند (Backend Logic)
#### [MODIFY] `backend/routes/pos.js`
- تعديل دالة البيع (`/sell`) لتستقبل طريقة الدفع (`paymentMethod`). وتقوم بإنشاء Transaction يحمل نفس طريقة الدفع.
#### [MODIFY] `backend/routes/cashier.js`
- جلب حركات اليوم وتجميعها لإنتاج: (إجمالي الكاش، إجمالي البنك، إجمالي المصروفات، صافي الدرج).

### 3. واجهة الكاشير (Frontend Cashier)
#### [MODIFY] `frontend/src/pages/cashier/CashierPOS.jsx`
- إضافة أزرار اختيار (كاش / Instapay) قبل الضغط على "إتمام البيع".
#### [NEW] `frontend/src/pages/cashier/CashierSafe.jsx`
- كروت منفصلة لـ (الرصيد النقدي بالدرج، أموال انستا باي).
- قائمة منسدلة بأنواع المصروفات عند السحب.
- زر "تقفيل الوردية" لإنهاء اليوم.

## 3. دليل الاستضافة والنشر (Deployment Guide)

نقترح البنية المجانية التالية الموثوقة والمجربة لإطلاق مشروعك بأقل تكلفة (0 دولار):

### 🌐 1. الواجهة الأمامية (Frontend - React)
* **المنصة**: **Vercel** أو **Netlify**
* **التكلفة**: **مجاني 100%**
* **المميزات**: ربط مباشر بـ GitHub، تحديث تلقائي عند تعديل الكود، دعم كامل للنظام وموقع سريع جداً بفضل شبكات الـ CDN.

### 🖥️ 2. السيرفر (Backend - Node.js + Express)
* **المنصة**: **Render.com** أو **Railway.app**
* **التكلفة**: **مجاني تماماً** (Render Free Tier)
* **تنبيه هام**: السيرفر المجاني على Render يدخل في وضع النوم (Sleep Mode) بعد 15 دقيقة خمول، مما يجعل أول طلب للموقع بعد الخمول يستغرق حوالي 30-50 ثانية للاستيقاظ.
* **البديل المدفوع الأرخص**: ترقية سيرفر Render إلى خطة **Render Starter** بتكلفة (7 دولار/شهرياً) للحفاظ على السيرفر شغالاً 24/7 بدون تأخير.

### 💾 3. قاعدة البيانات (Database - MongoDB)
* **المنصة**: **MongoDB Atlas**
* **التكلفة**: **مجاني 100%** (خطة Shared M0 Sandbox)
* **المميزات**: مساحة 512 ميجابايت (تكفي لملايين الطلبات والمنتجات الفواتير)، أمان عالٍ وسهولة ربط بمفتاح الاتصال (Connection String).
